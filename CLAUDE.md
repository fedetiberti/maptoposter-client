# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev          # Vite dev server on http://localhost:5173
npm run build        # tsc -b && vite build  → dist/
npm run typecheck    # tsc -b --noEmit (project-references build)
npm run lint         # eslint . (flat config, ignores dist)
npm run preview      # serve dist/
```

There is no test runner configured. `tsc` strictness is high (`strict`, `noUncheckedIndexedAccess`, `noUnusedLocals`, `noUnusedParameters`, `erasableSyntaxOnly`, `verbatimModuleSyntax`) — type errors will block the build. The `@/*` path alias is wired in both `tsconfig.app.json` and `vite.config.ts` and resolves to `src/*`.

`.npmrc` sets `legacy-peer-deps=true` for Vercel installs (React 19 / Vite 8 peer-dep mismatches).

## Architecture

Client-only SPA (no backend). Print-grade map poster designer built on MapLibre GL + OpenFreeMap vector tiles. State is a single `PosterState` reducer; everything else (the live map, the poster frame overlay, the export pipeline) is a pure projection of that state.

### Layered structure

`src/` follows a feature-folder + hexagonal layout. Each feature has up to four layers:

- `domain/` — pure types and pure functions (no React, no I/O). Examples: `poster/domain/PosterState.ts`, `theme/domain/Theme.ts`, `layout/domain/Layout.ts`.
- `application/` — orchestration: reducers, hooks, pipelines. May call services. Examples: `poster/application/posterReducer.ts`, `export/application/exportPipeline.ts`.
- `infrastructure/` — adapters that touch the outside world (network, MapLibre, canvas, localStorage). Examples: `map/infrastructure/MapEngine.ts`, `location/infrastructure/NominatimAdapter.ts`, `export/infrastructure/{png,pdf,svg}/`.
- `ui/` — React components.

`src/core/` holds cross-cutting ports/adapters (`IHttp`, `ICache`, `IFonts`) and a singleton `services` object that wires them. Adapters depend on the ports, never the other way around. New external dependencies should be introduced as a port in `core/ports.ts` plus an adapter, not by importing fetch/localStorage directly into application code.

`src/data/` holds bundled data tables: `themes.ts` (35 themes, ~26k of color spec), `layouts.ts` (27 print/social/wallpaper presets), `fonts.ts` (8 bundled `@fontsource` + ~50 lazy Google Fonts). These are the source of truth — adding a theme/layout/font means editing the corresponding `data/` file.

`src/shared/` holds small generic helpers (`haversine`, `useDebouncedValue`, `useResizeObserver`, color utils, `downloadBlob`).

### State model

The whole app reads from one `PosterState` (see `features/poster/domain/PosterState.ts`) — view, theme + per-key color overrides, layer toggles, title, font, markers, GPX, layout, export settings. Mutations go through `posterReducer` via `usePosterDispatch()`.

`PosterProvider` (`features/poster/application/PosterContext.tsx`) wires:

- **Boot order**: explicit `initialState` → URL hash share (`shareUrl.ts`, lz-string compressed) → `localStorage` (`persistState.ts`, key `mtp.poster.state.v1`, GPX stripped) → `DEFAULT_POSTER_STATE`.
- **Persistence**: every state change is debounced-saved (500ms) to localStorage.
- **Undo/redo**: ref-based history stacks (limit 50). Camera-only `SET_VIEW` changes still snapshot, but the history-push effect deliberately avoids reacting to other transient action types — read the comparison logic in `PosterContext.tsx` before tweaking. Undo/redo replay uses a synthetic `mtp:set-state` window event that dispatches a `HYDRATE` action; the reducer must keep the `HYDRATE` case in sync with the state shape.
- **Keyboard**: ⌘Z / ⇧⌘Z (or Ctrl+Y) bound globally, but skipped when focus is in an input/textarea/contenteditable.

### Live map ↔ state sync

`MapEngine` (`features/map/infrastructure/MapEngine.ts`) is a thin wrapper around `maplibregl.Map`. It uses a `suppressCallback` flag to avoid the classic camera-feedback loop: programmatic `setView` calls don't echo back through `onCameraChange`. When changing camera-related code, preserve this invariant — both directions are wired and breaking either causes oscillation or stuck cameras.

The map style is generated by `theme/application/mapStyleSpec.ts` from `(themeColors, layerToggles, tilesBaseUrl)`. The schema follows OpenMapTiles vector layers (`landcover`, `park`, `water`, `waterway`, `building`, `aeroway`, `transportation`). `setStyle(diff: true)` is used for live updates so layers don't flicker.

### Poster frame and export geometry (critical)

The right-edge dock pushes the on-screen poster frame off-center, so the geographic point at the **frame center** is **not** `state.view.lat/lon` (which tracks the *window* center). Two consumers must agree on this:

1. `PosterFrame.tsx` computes its on-screen rectangle via `layout/application/computePreviewBox.ts` (params: viewport size, layout aspect, `POSTER_MARGIN_PX`, `POSTER_RIGHT_DOCK_PX`).
2. `export/application/exportPipeline.ts → computeFrameProjection()` reprojects through web mercator using `lonLatToWorldPx` / `worldPxToLonLat` to make the export canvas show the same geographic extent as the live frame at a recomputed `exportZoom`.

If a recent commit mentions "export framing matches preview," it's this code path. Changing margin / dock width / aspect logic in one place requires the matching change in the other, or exports will drift from preview.

### Export pipeline

`runExport` in `exportPipeline.ts` orchestrates:

1. Resolve target pixel size from `LayoutSelection` (preset or custom) + DPI.
2. Validate against `SINGLE_PASS_MAX_SIDE` (8192) and `TILED_MAX_{SIDE,PIXELS}` (16384, 256MP).
3. Pre-load the title font via `services.fonts`.
4. Build a fresh `StyleSpecification` from theme + layer toggles.
5. Compute `exportView` + `exportZoom` (see frame-geometry note above).
6. Render the map: `singlePass.ts` uses one offscreen GL canvas; `tileRender.ts` stitches multiple passes when the long side exceeds 8192px. Both return a `{canvas, centerLat, centerLon, effectiveZoom}` result.
7. `compositeOverlay.ts` draws title block, markers, GPX track on top of the map canvas using the resolved projection.
8. Encode: `png/encodePng.ts` is a hand-rolled PNG that injects a `pHYs` chunk so print drivers honor the configured DPI (CRC32 in `png/crc32.ts`). `pdf/encodePdf.ts` is a hand-rolled minimal PDF 1.4. `svg/encodeSvg.ts` wraps the canvas in an SVG `<image>`.
9. `downloadBlob` triggers a browser download with a slugified filename.

Progress is reported through a `ProgressCb` with stages `preparing → rendering → compositing → encoding → downloading → done`.

### Fonts

`data/fonts.ts` has two source kinds: `bundled` (`@fontsource*` packages, eagerly available via `styles/fonts.css`) and `google` (loaded on demand). `core/fonts/ensureGoogleFont.ts` injects a Google Fonts `<link>` and waits on `document.fonts.load(...)`. The export pipeline calls `services.fonts.ensureLoaded(...)` before rasterizing so titles don't render with a fallback face.

### PWA and runtime caching

`vite.config.ts` configures `vite-plugin-pwa` with Workbox runtime caching: OpenFreeMap tiles (CacheFirst, 30d), Nominatim (StaleWhileRevalidate, 7d), Google Fonts CSS (SWR) and font files (CacheFirst, 1y). `devOptions.enabled: false` — the service worker only runs in production builds. `features/install/` handles the install prompt and update prompt UX.

### Environment

Two optional Vite env vars override defaults at build time:

- `VITE_NOMINATIM_BASE_URL` (default `https://nominatim.openstreetmap.org`)
- `VITE_TILES_BASE_URL` (default `https://tiles.openfreemap.org`)

Both are read in `core/services.ts` and threaded through adapters / style spec — never read `import.meta.env` directly from feature code.
